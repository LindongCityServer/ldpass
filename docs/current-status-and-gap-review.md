# 临东通现状与差距评审

本文档记录截至当前原型测试后的产品现状、已发现问题、后续开发优先级和仍需拍板的问题。它不是替代需求文档，而是给后续迭代用的“作战清单”。

## 1. 当前总体判断

项目已经不再只是静态页面骨架，后端已经覆盖账号、会话、用户审核、发卡方入驻、模板审核、发放领取码、钱包领卡、卡券详情、权益流水、核销、冲正、转赠、争议、审计、主题计划、存储监控、协议文档和开放 API 的一批最小闭环。

从可用产品角度看，当前已经进入“最小闭环可跑、下一轮继续打磨”的阶段：

- 后台信息架构已开始从“临时表单”收敛为统一后台壳：管理员后台和发卡方后台已有统一标题栏与导航，后续可继续按页面复杂度拆分二级面板和局部组件。
- 一些关键业务语义已经开始收敛：核销里的服务器账号确认不再等同于“账号已验证”，而是每次操作重新发放聊天验证码；主题计划也已经改为按具体日期时间生效的计划列表。
- 卡券详情已经完成第二版拆层：详情默认展示摘要和模块入口，卡片详情、充值、选择来源卡、发起核销、待确认核销、交易记录、争议记录、提交争议和转赠都按弹窗闭环打开，避免手机和平板视图被信息压垮。

### 1.1 当前闭环地图

给后续重新画原型图时，可以先按这些闭环切页面和弹窗：

- 普通用户注册与登录闭环：用户注册或服务器验证注册 -> 管理员审核或服务器聊天验证码通过 -> 创建会话 -> 进入钱包；待审核、被拒、封禁和已删除状态都在账户页走受限视图。
- 账户与设备安全闭环：账户页标题栏进入设置项 -> PIN、服务器账号换绑、提醒偏好、设备管理、新设备确认、审核资料补充和注销都在弹窗中完成 -> 成功后刷新账户状态或撤销相关会话。
- 领卡闭环：管理员或发卡方生成领取码/添加链接/二维码 -> 用户打开 `/add` 或扫码 -> 未登录先登录并回跳 -> 预览卡券 -> 确认领取 -> 钱包首页出现对应卡片。
- 钱包详情闭环：钱包首页选择卡片 -> 详情摘要显示余额、发卡方模块、操作入口和交易入口 -> 卡片详情、发卡方、额度补充、来源卡选择、发起核销、待确认核销、交易记录、争议记录、提交争议和转赠均由弹窗承接。
- 卡面与模板闭环：发卡方创建或提交新版模板 -> 卡面预览只展示卡面视觉和卡号末四个字符，标题、Logo、发卡方和规则说明移到卡外 -> 管理员审核 -> 发卡时绑定已审核版本；管理员可同时查看待审和已过审模板。
- 发卡闭环：发卡方选择已审核模板 -> 单个或批量生成领取码 -> 后台同时展示卡号、掩码卡号、领取码和添加链接 -> 后续列表只保留领取码尾号，可撤销或作废重发未领取码。
- 发卡方核销闭环：发卡方登录后进入 `/provider/redemptions` -> 输入已领取卡片的完整卡号 -> 后端按 `publicNumber` 定位卡片 -> 校验卡券已领取、状态和允许核销名单 -> 发起待用户确认核销。
- 用户确认核销闭环：用户在钱包详情查看待确认核销 -> 选择 PIN 或本次服务器聊天验证码确认 -> 成功后扣减权益、写流水和事件；失败、过期、取消和冲正都有状态和审计记录。
- 发卡方资料审批闭环：Active 发卡方提交名称、头像图床 HTTPS 图片链接、介绍链接、联系人、联系邮箱和业务说明变更 -> 管理员审核 -> 通过后才写入正式 Provider 资料；头像可留空，填写时必须是 HTTPS 图片链接。
- 开放集成闭环：发卡方提交 API 密钥或 Webhook 配置申请 -> 管理员审核 -> 发卡方一次性查看明文密钥或签名密钥 -> 开放 API / Webhook 复用同一套发卡、卡券、核销、操作链接和审计事件。

## 2. 已实现能力清单

### 2.1 用户侧

- 用户注册：支持管理员审核注册和服务器验证注册两个入口；注册时记录 IP 与本地粗分类 IP 属地，管理员审核页可查看；被拒绝用户可在账户页查看原因并重新提交审核信息；服务器验证注册成功后会直接创建当前浏览器会话并进入钱包。
- 用户登录：支持用户名优先、邮箱备选，登录态通过 HttpOnly Cookie 保存；待审核、被拒绝、等待服务器验证或封禁用户登录后进入受限账户页。
- 设备安全：支持设备绑定、新设备服务器验证或已登录设备确认、服务器账号换绑、PIN 设置与重置。
- 钱包首页：支持分类、搜索、编辑排序、归档、卡片详情、离线基础快照；首页卡片优先展示已审核卡面图片和左下角卡号末四个字符，没有卡面图时才显示标题和余额，模板隐藏标题时只保留卡号末四个字符。
- 添加卡券：支持领取码、添加链接、二维码入口，领取前预览，未登录后回跳继续领取；`/add` 只承担普通用户添加卡券流程，不再集成发卡方核销模式。
- 卡券详情：支持余额/权益、流水、票券字段、位置核验、转赠、争议、用户主动发起消耗。
- 核销确认：支持 PIN 确认和服务器账号确认；服务器账号确认已改为本次重新发放 `LDPASS-` 前缀验证码后再检查服务器聊天；BDSLM 聊天消息读取已兼容 `name` / `text` 和 `content` / `message` / `body` 字段。
- 账户页：已改为标题栏布局，主视图展示账户摘要和设置入口；退出登录、注销账户、PIN、服务器账号换绑、提醒设置、设备列表、新设备确认和注册审核补充通过弹窗处理；普通用户头像由已验证服务器账号派生 Minecraft 头像链接，非 Active 用户只显示账户状态、退出、注销和注册审核补充入口。

### 2.2 发卡方侧

- 发卡方公开入驻申请和发卡方登录；被拒申请可用原邮箱、原标识和负责人密码修改资料后重新提交；管理员可停用、恢复或归档发卡方。
- Active 发卡方可以在工作台提交资料变更申请，当前覆盖提供方名称、头像图床 HTTPS 图片链接、介绍链接、联系人、联系邮箱和业务说明；管理员审核通过前不会改写正式资料。头像可留空，如填写必须是 HTTPS 且路径后缀为 `png`、`jpg`、`jpeg`、`webp`、`gif` 或 `avif`；介绍链接可留空，如填写必须是 HTTPS URL。
- 模板创建、新版提交、管理员审核后激活；模板规则可配置允许核销方名单，默认只允许发卡方自身核销。
- 单个/批量生成领取码和添加链接。
- 查看自己发放的卡券，调整金额/积分/次数，导出 CSV。
- 发起消耗请求、在 `/provider/redemptions` 按已领取卡片卡号发起核销、取消待确认请求、冲正已完成核销。
- 票券字段变更审批：发卡方提交票券字段变更后进入管理员审核，审核通过才写入卡券可见信息。
- API 密钥创建审批、开放发放/查询/调整/核销接口。
- Webhook 端点创建、修改、启停、删除和密钥轮换审批、一次性签名密钥、投递记录和失败重试。

### 2.3 管理员侧

- 超级管理员种子脚本，管理员登录需要密码 + PIN。
- 用户审核、拒绝、用户目录、注册 IP / IP 属地查看、用户目录 CSV 导出、管理员介入重置用户密码。
- 发卡方手动创建、入驻审核、拒绝、提供方目录搜索与 CSV 导出。
- 提供方资料变更审核：管理员可查看发卡方提交的资料变更，对比当前值和提交值；名称、头像、介绍链接、联系人、邮箱和业务说明都必须审核通过后才应用到 Provider 正式资料，拒绝时保留审核意见。
- Webhook 配置审核：发卡方新增、修改、启停、删除回调端点和轮换签名密钥都会先进入待审申请；管理员通过后才写入正式端点配置；新增和轮换产生的签名密钥只允许发卡方一次性查看。
- 发卡方停用、恢复和归档；停用会撤销负责人会话，归档会停用 API 密钥和 Webhook 端点。
- 模板审核、卡面模板变体管理。
- 管理员生成领取码。
- 全站卡券查询、权益调整、冻结/解冻、CSV 导出。
- 票券字段变更审核，通过后更新卡券可见票券信息。
- 争议状态流转。
- 后台概览：汇总用户、提供方、提供方资料变更、Webhook 配置、模板、票券字段变更、争议、存储告警、卡券和操作链接关键数字；最近审计记录只在审计日志页查看。
- 审计日志查询与按当前筛选条件导出 CSV。
- 全站公告和维护状态配置。
- 按 UTC+8 绝对生效时间配置主题计划。
- 磁盘剩余空间检测、项目占用统计和告警。
- 服务条款、隐私政策的维护入口；提供方协议第一阶段已从产品入口、公开页面和管理员维护入口去掉，仅保留历史预留 key。

管理员后台信息架构对照：

- `/admin`：关键数字和待办；待办只在数量大于 0 时展示处理入口，最近审计只在审计日志页查看。
- `/admin/users`：搜索、状态筛选、刷新、导出 CSV；用户列表展示头像占位、用户名、ID、邮箱、状态和详情、审核、重置密码、封禁/解封、注销操作。
- `/admin/providers`：发卡方、资料、API、Webhook 四段切换；发卡方目录提供搜索、刷新、导出 CSV、详情、停用/启用、归档，待审队列提供对应审核操作。
- `/admin/pass-templates`：搜索、分类筛选、刷新、导出 CSV；同一模板合并展示多个版本，每个版本保留详情、驳回/打回、批准操作。
- `/admin/card-template-variants`：新增模板变体弹窗、搜索、刷新、导出 CSV；列表展示标识、名称、分类、启停状态和详情/编辑操作。
- `/admin/add-pass-token`：生成领取码弹窗；提交后弹窗展示完整领取码、复制和二维码，并提示完整码只能查看一次；列表提供搜索、状态筛选、分类筛选、刷新、重置、导出 CSV、查看卡券、撤销、作废并重发。
- `/admin/passes`：卡券列表和票券变更分段切换；卡券列表提供搜索、刷新、导出卡券/流水 CSV、详情、调整余额、冻结/解冻、补充冲正；票券变更队列提供拒绝和通过。
- `/admin/client-applications`：登记新应用弹窗、搜索、刷新、导出 CSV；列表提供详情/编辑。
- `/admin/disputes`：搜索、状态筛选、刷新、导出 CSV；列表提供详情、处理、补充冲正和核销冲正。
- `/admin/audit`：事件类型、操作者 ID、对象 ID 筛选，刷新和导出 CSV；摘要字段以中文语义化键值展示。
- `/admin/platform`：公告、维护、主题三段切换；公告和维护状态保存，主题计划以内嵌组件维护。
- `/admin/legal`：服务条款和隐私政策分段切换，保留预览公开页和保存文档操作。
- `/admin/storage`：存储状态、项目占用明细和活动告警，保留立即检测操作。

发卡方后台信息架构对照：

- `/provider/dashboard`：发卡方摘要、详情、资料变更、退出；移动端作为上级导航页展示业务和接入入口。
- `/provider/templates`：新建模板弹窗、模板列表、模板详情弹窗；已过审模板保留随时打回能力，规则摘要以中文语义化展示。
- `/provider/issue`：发放领取码弹窗；单个、批量和作废并重发结果都在弹窗中展示完整领取码，并提示完整领取码只能查看一次，关闭后列表只保留尾号。
- `/provider/redemptions`：独立核销页，按已领取卡片的完整卡号定位并发起核销。
- `/provider/passes`：卡券列表展示摘要和并列操作；详情、调整余额、发起消耗、操作链接、状态管理、票券字段和票券变更记录通过弹窗承接。
- `/provider/api-keys`：提交 API 密钥申请弹窗；列表展示摘要，权限、有效期和使用状态进入详情弹窗；明文密钥只允许一次性弹窗查看。
- `/provider/webhooks`：提交回调端点申请弹窗；列表展示摘要，URL、事件、投递状态和错误进入详情弹窗；修改配置表单通过弹窗提交。

### 3.6 服务器聊天验证

需要把所有服务器验证码消息加前缀，避免用户在聊天里输入一串孤立数字时被误解，也降低误触发概率。

建议默认格式：

```text
LDPASS-123456
```

验证规则：

- 网站展示完整带前缀验证码。
- 用户必须在服务器聊天中输入完整内容。
- 如果同一个服务器 ID 发送了其他内容，当前验证码应失效并刷新。
- 前缀需要进入哈希校验前的规范化流程，避免前端显示 `LDPASS-123456`、后端只校验 `123456` 造成歧义。

服务器账号确认已完成第一版调整：核销请求会先调用 `POST /api/wallet/redemption-requests/:requestId/server-challenge/start` 创建本次 `pass_use` 验证码，用户用绑定的服务器 ID 在服务器聊天中输入完整验证码后，再调用 `POST /api/wallet/redemption-requests/:requestId/confirm-server` 检查并执行扣减。

### 3.7 领取码展示规则

当前完整领取码只在生成界面展示，后续如果只能看到掩码或状态，这是合理的安全设计：领取码数据库应只保存哈希，不能反推出原始领取码。

当前生成规则已经收紧为 `LD-` 前缀后只包含大写字母和数字。领取/预览入口不应把这个新生成规则作为强校验条件，因为上线测试期间曾生成过带连字符或下划线的历史码；接口应继续 trim、转大写并按哈希查找，保证旧码可被领取或作废处理。

当前已完成第一版后台列表：

- 管理员 `/admin/add-pass-token` 可以查看最近领取码记录，按尾号、提供方、模板、领取人和状态筛选。
- 发卡方 `/provider/issue` 可以查看自己生成的领取码记录，按尾号、模板、领取人和状态筛选。
- 管理员和发卡方都可以撤销仍为 `Active` 的领取码；已领取、已过期或已撤销的领取码不能再次撤销。
- 管理员和发卡方都可以对未领取且仍关联 `Issued` 卡券的领取码执行作废并重发；新完整领取码只在重发结果中展示一次，旧码会置为 `Revoked`。
- 开放 API 已补齐发卡方领取码列表、撤销和作废并重发接口，分别需要 `add_pass_token:read`、`add_pass_token:revoke`、`add_pass_token:reissue` 权限。
- 撤销会发布 `AddPassTokenRevoked` 事件，重发会发布 `AddPassTokenReissued` 事件并进入审计链路。

建议产品规则固定为：

- 生成成功后一次性展示完整领取码和完整链接。
- 后续列表只展示尾号、创建时间、状态、领取人、过期时间。
- 如果完整码丢失，只允许作废旧码并重新生成；已领取的码不能重发，需要走卡券状态、转赠或争议流程。
- 不建议为了“后台可再次查看完整码”而明文存储；如确实需要二次查看，只能改为服务端加密存储，并增加访问审计和二次验证。

### 3.8 卡内额度补充

你提出的“从其他支持对外补充的卡片内扣减余额，补充到目标卡片”本质上不是资金充值，而是平台内两张卡之间的权益转移。建议产品文案使用：

- 目标卡：补充额度、额度补充。
- 来源卡：消耗来源卡额度、用于补充。
- 结果页：补充成功、来源卡已消耗、目标卡已增加。

当前已完成第一版同步补充能力：

- 发卡方模板规则新增 `allowTopUpIn` 和 `allowTopUpOut`。
- 用户在钱包详情中选择来源卡、补充值和确认方式。选择 PIN 时调用 `POST /api/wallet/passes/:passId/top-ups`；选择服务器账号确认时先调用 `POST /api/wallet/passes/:passId/top-ups/server-challenge/start` 生成本次 `LDPASS-` 验证码，再调用 `POST /api/wallet/passes/:passId/top-ups/confirm-server` 检查服务器聊天并完成补充。
- 第一版只允许同一用户持有的两张 `Active` 卡券之间操作。
- 来源卡和目标卡必须是同一权益类型，且来源卡必须开放“作为额度补充来源”，目标卡必须开放“被其他卡补充额度”。
- 后端会先校验用户 PIN 或本次服务器聊天验证码。PIN 路径发布 `PinVerificationSucceeded(purpose: pass_top_up)`；服务器账号路径发布 `ServerVerificationCodeIssued/Rotated(purpose: pass_top_up)`。校验通过后再使用同一个事务同时扣减来源卡、增加目标卡，并写入两条 `LedgerEntry`，原因均为 `top_up`。
- 用户主动钱包补充和补充链接都会先创建 `PassTopUpRequest`：PIN 路径从 `Created` 直接进入 `Succeeded` 或 `Failed`；服务器账号路径先进入 `WaitingVerification`，验证码过期进入 `Expired`，用户可取消进入 `Cancelled`；来自补充链接的请求会记录 `actionLinkId`。
- 成功后发布 `PassTopUpRequested`、`PassTopUpSucceeded` 和两条 `PassBalanceChanged`，失败、过期、取消分别发布 `PassTopUpFailed`、`PassTopUpExpired`、`PassTopUpCancelled`；发卡方 Webhook 可订阅这些额度补充事件。
- 用户钱包详情会通过 `GET /api/wallet/top-ups?passId=...` 展示该卡相关补充记录，记录包含来源卡、目标卡、状态、确认方式、失败原因和关键时间点；用户可从补充记录直接提交 `pass_top_up` 争议。
- 管理员可在 `/admin/passes` 输入补充 ID、冲正原因和管理员 PIN，对已完成额度补充执行反向冲正。系统不会删除原 `top_up` 流水，而是追加两条 `refund` 流水，把来源卡额度加回、目标卡额度扣回，并发布 `PassTopUpReversed` 与两条 `PassBalanceChanged`。
- 管理员冲正成功后会把对应 `PassTopUpRequest` 状态更新为 `Reversed`。

当前补充请求模型已经覆盖待确认、可取消、过期、失败和冲正状态：

```ts
interface PassTopUpRule {
  allowTopUpIn: boolean; // 本卡是否允许被补充
  allowTopUpOut: boolean; // 本卡是否允许作为来源卡
  acceptedSourceProviderIds?: string[];
  acceptedSourceTemplateIds?: string[];
  minValue?: string;
  maxValue?: string;
}

interface PassTopUpRequest {
  id: string;
  sourcePassId: string;
  targetPassId: string;
  actionLinkId?: string;
  requestedValue: string;
  status: 'Created' | 'WaitingVerification' | 'Succeeded' | 'Failed' | 'Cancelled' | 'Expired' | 'Reversed';
  verificationMethod: 'pin' | 'server_account';
}
```

关键约束：

- 来源卡和目标卡都必须明确开启对应能力。
- 必须使用事务同时写入两张卡流水：来源卡扣减、目标卡增加。
- 第一版已使用服务端生成的补充 ID 写入两条唯一流水键，钱包补充和补充链接都会复用该补充 ID，避免重复确认写入多组流水。
- 反向冲正第一版已由管理员按补充 ID 执行；用户可把补充历史中的额度补充请求作为 `pass_top_up` 争议对象提交；“目标卡额度不足时是否允许冲到负数”属于后续产品决策项，当前实现保持不透支。
- 跨用户补充、跨发卡方结算属于后续产品决策项，当前实现限制在同一用户名下卡券。

### 3.9 链接扣减与链接补充

当前已完成第一版“操作链接”能力，区别于添加卡券链接：

- 使用链接：外部项目生成一次性链接，用户登录后预览本次将消耗的卡券、数值、提供方和有效期，再通过 PIN 或服务器聊天验证码确认。
- 补充链接：外部项目或发卡方生成一次性链接，用户选择来源卡，预览来源卡扣减和目标卡增加，再通过 PIN 或服务器聊天验证码确认。

已实现入口：

- 发卡方后台 `/provider/passes` 针对已领取卡券生成操作链接，完整链接只在生成结果中展示一次。
- 发卡方后台 `/provider/passes` 可以按类型、状态和当前/全部卡券筛选操作链接，并撤销单条或批量撤销尚未使用且未过期的链接。
- 开放 API `POST /api/open/provider/action-links` 可通过 `action_links:create` scope 生成操作链接，`GET /api/open/provider/action-links` 可通过 `action_links:read` scope 查询操作链接，单条撤销和批量撤销分别通过 `POST /api/open/provider/action-links/:actionLinkId/revoke`、`POST /api/open/provider/action-links/revoke-batch` 并使用 `action_links:revoke` scope。
- 用户打开 `/action?token=...` 后必须是已登录 Active 用户，先读取预览，再确认。
- 服务器聊天验证码和 PIN 验证目的已区分 `pass_use` 和 `pass_top_up`，审计上能区分确认使用与额度补充。
- 链接 token 数据库只保存哈希；链接状态包含 `Active`、`Consumed`、`Expired`、`Revoked`。
- 后端读取列表、用户打开链接或后台清理任务扫描到过期链接时，会把过期链接标记为 `Expired`。
- API 进程内置操作链接过期清理器，默认每 60 秒扫描一次，支持通过 `ACTION_LINK_EXPIRY_SWEEP_*` 环境变量关闭、调间隔或调批量大小。
- 审计和发卡方 Webhook 已支持 `WalletActionLinkCreated`、`WalletActionLinkConsumed`、`WalletActionLinkExpired`、`WalletActionLinkRevoked`；补充链接还会复用 `PassTopUpRequested/Failed/Expired/Cancelled/Succeeded/Reversed` 状态事件。

当前争议处理规则已补齐第一版：

- 补充链接已接入请求状态机、取消、过期、失败、管理员冲正、争议对象关联和用户侧补充历史列表；管理员处理核销请求或额度补充争议时可在争议页直接执行冲正并标记已反转。
- 普通卡券、流水和管理员调整争议只允许按状态机推进处理结论；需要补充、认可、驳回、关闭都必须填写处理备注，不能直接伪装成账务反转。
- 后端创建争议时会校验核销请求、额度补充请求、流水或管理员调整记录确实属于当前用户持有的卡券，避免用户构造其他对象 ID 发起争议。

### 3.10 发卡方集成配置审批

当前已把 Webhook 端点创建、修改、启停、删除和密钥轮换纳入管理员审批：

- 发卡方在 `/provider/webhooks` 提交端点名称、回调地址、订阅事件和申请说明。
- 新增端点申请通过前不会创建可投递端点，也不会下发签名密钥；通过后发卡方可在 `/provider/webhooks` 一次性查看签名密钥。
- 修改端点、启停端点、删除端点和轮换签名密钥都会创建 `ProviderWebhookChangeRequest`，状态为 `PendingReview`；管理员通过前正式 `ProviderWebhookEndpoint` 不变。
- 管理员在 `/admin/providers` 的“Webhook 配置待审”区域查看申请类型、目标端点、目标状态和事件范围，通过后才应用修改、启停、软删除端点或写入新签名密钥。
- Webhook 删除采用软删除：端点从发卡方工作列表和投递调度中移除，但保留投递记录、申请记录和审计引用。
- 新增和轮换产生的签名密钥查看后，服务端会清空申请表中的密钥密文；第二次查看同一申请的签名密钥会被拒绝。
- 通用提交、通过、拒绝会分别写入 `ProviderWebhookChangeSubmitted`、`ProviderWebhookChangeApproved`、`ProviderWebhookChangeRejected` 审计事件；新增端点仍保留 `ProviderWebhookEndpointCreateSubmitted`、`ProviderWebhookEndpointCreateApproved`、`ProviderWebhookEndpointCreateRejected` 兼容事件；实际创建、修改、轮换、删除分别写入 `ProviderWebhookEndpointCreated`、`ProviderWebhookEndpointUpdated`、`ProviderWebhookSecretRotated`、`ProviderWebhookEndpointDeleted`；一次性查看密钥写入 `ProviderWebhookSecretClaimed`。

当前也已把 API 密钥创建、轮换和停用纳入管理员审批：

- 发卡方在 `/provider/api-keys` 提交密钥创建、轮换或停用申请，申请中包含目标密钥、权限范围、有效期和申请说明。
- 系统创建 `ProviderApiKeyChangeRequest`，状态为 `PendingReview`；创建/轮换申请不会立即创建可调用开放 API 的新密钥，停用申请不会立即让旧密钥失效。
- 管理员在 `/admin/providers` 的“API 密钥待审”区域查看申请类型和内容，重点检查 scope 范围是否过大、轮换/停用目标是否合理。
- 管理员通过创建/轮换申请后才创建 `ProviderApiKey`，真实密钥表仍只保存 token 哈希；通过停用申请后才写入目标密钥的 `revokedAt`。
- 发卡方在 `/provider/api-keys` 可一次性查看创建/轮换产生的新明文 API 密钥；查看后服务端清空申请表中的密钥密文，只保留 `plainApiKeyViewedAt`。
- 第二次查看同一申请的明文 API 密钥会被拒绝；如密钥丢失，后续仍通过新的“轮换密钥申请”生成新密钥。
- 通用提交、通过、拒绝会分别写入 `ProviderApiKeyChangeSubmitted`、`ProviderApiKeyChangeApproved`、`ProviderApiKeyChangeRejected` 审计事件；一次性查看、实际创建、实际轮换、实际停用会分别写入 `ProviderApiKeySecretClaimed`、`ProviderApiKeyCreated`、`ProviderApiKeyRotated`、`ProviderApiKeyRevoked`。创建申请仍保留 `ProviderApiKeyCreateSubmitted`、`ProviderApiKeyCreateApproved`、`ProviderApiKeyCreateRejected` 兼容事件。

后续可继续细化的集成配置规则：

- 更细的“哪些 scope / 哪些 Webhook 事件需要强制审批、哪些可以走低风险快速审批”属于产品和安全策略决策项；当前实现采用统一审批，优先保证安全闭环。

## 5. 需要你拍板的问题

1. 管理员删除用户已采用软删除。
   当前结论：状态变为 `Deleted`，保留卡券、流水、审计、争议引用，避免硬删除破坏可追溯性。

2. 用户封禁是否允许登录到账户页？
   建议：允许登录后只展示封禁状态、退出登录、注销账户和申诉说明；不允许使用卡包、核销、转赠、设备设置。

3. 服务器验证码前缀已采用 `LDPASS-`。
   当前规则：用户需要在服务器聊天内发送完整验证码，例如 `LDPASS-123456`，便于审计和降低误匹配概率。

4. 多位置核验第一阶段是否只要“命中任意一个位置即通过”？
   建议：是。`all` 模式很容易让配置变复杂，第一阶段保留数据结构即可。

5. 卡内额度补充第一阶段已限制为同一用户名下卡券。
   后续待拍板：是否允许跨用户、跨发卡方、指定来源模板，以及对应争议和授权规则。

6. 完整领取码是否坚持只展示一次？
   建议：只展示一次。后续后台提供作废和重新生成，不提供二次明文查看。

7. 主题计划没有已生效条目时，自动主题色兜底到哪一种？
   建议：兜底青绿色，但用户默认选项仍是“自动切换”；这是平台配置缺失时的技术兜底，不是用户默认选青绿色。

8. 服务条款和隐私政策是否允许先以“平台说明框架”发布？
   建议：允许。先覆盖账号、卡券、服务器聊天验证、设备绑定、日志审计、非真实支付声明、用户注销和数据保留。

9. Webhook 修改、删除、密钥轮换是否都要进入同一个“集成配置申请”队列？
   当前结论：是。创建、修改、启停、删除和密钥轮换已进入同一个 `ProviderWebhookChangeRequest` 队列，管理员通过后才应用正式配置。
